@Target

@Target用来表示注解作用范围，超过这个作用范围，编译的时候就会报错。

枚举类如上所示，其中包括

枚举变量 作用域
Type 类或接口
FIELD 域
METHOD 方法
PARAMETER 参数
CONSTRUCTOR 构造方法
LOCAL_VARIABLE 局部变量
ANNOTATION_TYPE 注解类型（被interface修饰的类型）
PACKAGE 用于记录java文件的package信息
这里这个@Inner注解可以使用在方法和类或接口上。

@Retention

@Rentention定义了该Annotation被保留的时间长短

@Rentention由RetentionPolicy表示需要在什么级别保存该注解信息，用于描述注解的生命周期。

枚举变量 有效范围
SOURCE 在源文件中有效
CLASS 在class文件中有效
RUNTIME 在运行时有效

这边的@Inner注解表示在运行时有效。

@Documented

Documented注解表明这个注释是由 javadoc记录的，在默认情况下也有类似的记录工具。 如果一个类型声明被注释了文档化，它的注释成为公共API的一部分。

@Component

Component泛指组件，当组件不好归类的时候，我们可以使用这个注解进行标注

二分法

今日做LeetCode剑指Offer，做到了二分法的题目，写起来老是自己绕进去，看一看整理一下基本模板。

例题

剑指 Offer 53 - II. 0～n-1中缺失的数字

正所谓，看到递增数组，就要用二分法，用遍历方法等于白做。

我的第一思路是，取low=0,high=n-1，然后开始二分mid = low + (high - low) / 2，验证如果nums[mid] != mid，则证明这个数字在low到mid范围内，否则是在mid到high范围内。

二分法最蛋疼的一点就是这个边界，我这个解法在每次都是令low或high=mid，然后结束循环的条件是low < high -1，避免low = high -1导致mid无限等于low。最终就会出现low与high差2，比如对一个递增数字找到某个数字时，low与high中间的值就是需要的结果。这个题目实际上应该等于nums[mid]-1，但是我这个mid放循环里了，所以就用的nums[low]+1。

LeetCode题解区我看到了一个非常简洁的二分

属实牛批。

模板

模板1

二分将范围分成(low,mid)和(mid+1,high)，更新操作时令low = mid + 1，high = mid，判断条件使用low < high即可，这个是用得最广泛的，最终low与high塌成一个点。

模板2

二分将范围分成(low,mid-1)和(mid,high)时，更新操作时令low = mid， high = mid - 1，mid应该设为(low + high + 1) / 2

模板3

也就是我用的那个，将范围分成(low,mid)和(mid,high)，边界已经排除，这方法就是得使劲排除边界条件。

这里有一个蓄水池算法，是我抄的，学到了学到了，用在不知道总数是多少的情况。**抽奖算法**

所谓蓄水池算法，从头来说可以这样形容：

（1）来到第一个节点，池子里只有一个数字，这时候随机到这个数字的概率必为100%

（2）来到第二个节点，这时候池子里有两个数字，这时候随机到这两个数字中其中一个的概率为50%

（3）来到第三个节点，这时候池子里有三个数字，这时候随机到第三个数字的概率为33%，随机到前两个数字的概率均为66%*50%=33%，三个数字概率相同。

以此类推，链表遍历完以后，能够保证随机到任意一个数字的概率均相等。

本地事务和分布式事务

发表于 2022-01-04

本地事务

今天工作时涉及到了事务这个概念，以前没有真正接触过，趁这个机会学习一下事务这个概念，并且联系一下经常听到的分布式事务，读一下资料，记一下学到的知识点。这一篇先大体抄一点本地事务的基本知识点。

事务

事务是数据库操作的最小工作单元，是作为单个逻辑工作单元执行的一系列操作；这些操作作为一个整体一起向系统提交，要么都执行、要么都不执行；事务是一组不可再分割的操作集合（工作逻辑单元）；

PS：抄概念，手打一遍，哈哈，希望能记住点。

事务的特性

原子性（atomicity）：不可分割，事务中的操作要么都做，要么都不做。

一致性（consistency）：事务必须是使数据库从一个一致性状态变到另一个一致性状态，事务的中间状态不能被观察到。

隔离性（isolation）：一个事务不能被其他事务干扰，事务内部的操作及使用数据对并发的其它事务是隔离的。隔离性分为四个级别：读未提交（read uncommitted）、读已提交（read committed）、可重复读（repeatable read）、串行化（serializable）。

持久性（durability）：事务一旦提交，它对数据库中数据的改变就应该是永久性的。

JAVA几种类型的事务

JAVA事务类型有三种：JDBC事务、JTA事务、容器事务。

JDBC事务

在JDBC中处理事务，都是通过Connection完成的。同一事务中所有的操作，都在使用同一个Connection对象。JDBC事务默认是开启的，并且是默认提交。

传统JDBC操作流程：

1).获取JDBC连接 2).声明SQL 3).预编译SQL 4).执行SQL 5).处理结果集 6).释放结果集 7).释放Statement 8).提交事务 9).处理异常并回滚事务 10).释放JDBC连接

JDBC优缺点：1.冗长、重复 2.显示事务控制 3.每个步骤不可获取 4.显示处理受检查异常

JDBC为使用Java进行数据库的事务操作提供了最基本的支持。通过JDBC事务，我们可以将多个SQL语句放到同一个事务中，保证其ACID特性。JDBC事务的主要优点就是API比较简单，可以实现最基本的事务操作，性能也相对较好。但是，JDBC事务有一个局限：一个 JDBC 事务不能跨越多个数据库！所以，如果涉及到多数据库的操作或者分布式场景，JDBC事务就无能为力了。

JTA事务

JTA(Java Transaction API)提供了跨数据库连接（或其他JTA资源）的事务管理能力。JTA事务管理则由JTA容器实现，J2ee框架中事务管理器与应用程序，资源管理器，以及应用服务器之间的事务通讯。

JTA的优缺点：

JTA的优点很明显，就是提供了分布式事务的解决方案，严格的ACID。但是，标准的JTA方式的事务管理在日常开发中并不常用。

JTA的缺点是实现复杂，通常情况下，JTA UserTransaction需要从JNDI获取。这意味着，如果我们使用JTA，就需要同时使用JTA和JNDI。
