Lombok的几个注解

一、Lombok引入

IDEA开发中，Lombok需要两部分，一个是maven依赖，一个是IDEA插件

1、maven依赖

2、IDEA插件

二、Lombok注解

参考：Lombok常用注解 - 草木物语 - 博客园 (cnblogs.com)

1、@Getter/@Setter

自动生成get和set方法

2、@ToString

重写toString()方法，打印出所有变量

3、@EqualsAndHashCode

自动生成equals(Object obj)方法和hashCode()方法，包括所有非静态变量和非 transient 的变量。

排除某变量：@EqualsAndHashCode(exclude = “xxx”)

4、@NoArgsConstructor, @AllArgsConstructor, @RequiredArgsConstructor

构造器三兄弟，区别在于生成构造器的参数不同。

@NoArgsConstructor：生成无参构造器

@AllArgsConstructor：生成包含所有参数的构造器

@RequiredArgsConstructor：生成一个包含 “特定参数” 的构造器，特定参数指的是那些有加上 final 修饰词、还有@NonNull修饰的变量们，如果没有这种变量的话，就会生成一个无参构造。

ATTENTION：如果没有显示写出构造器，编译器会自动帮我们写一个无参构造器，如果有构造器了，编译器就不会给我们生成了。如果用@AllArgsConstructor去修饰类，会生成一个全参构造器，这时候无参构造器就没了。因此用@AllArgsConstructor，一定要补一个@NoArgsConstructor。

5、@Data

这位是重量级，大杂烩，包含了@Getter、@Setter、@ToString、@EqualsAndHashCode、@RequiredArgsConstructor。

@Data 是使用频率最高的 lombok 注解，通常 @Data 会加在一个值可以被更新的对象上，像是日常使用的 DTO 们、或是与数据库交互的 Entity 们，就很适合加上 @Data 注解，也就是 @Data for mutable class

6、@Value

也是整合包，但是他会把所有的变量都设成 final 的，其他的就跟 @Data 一样，等于同时加了以下注解：@Getter (注意没有setter)、@ToString、@EqualsAndHashCode、@RequiredArgsConstructor。

适合加在值不希望被改变的类上，像是某个类的值当创建后就不希望被更改，只希望我们读它而已，就适合加上 @Value 注解，也就是 @Value for immutable class。

ATTENTION：lombok 的注解 @Value 和另一个 Spring 的注解 @Value 撞名，在 import 时不要 import 错了

7、@Builder

自动生成流式 set 值写法。这个还没怎么用过。

加上这个注解以后，可以使用流式，如：Person.builder().setName(“xxx”).setAge(“19”).build()。

还有另外的一种注解@Accessors，可以用chain = true或fluent = true。

chain = true，生成setter返回this。fluent = true，生成的方法不带get前缀（不推荐），容易寄。

这样可以这样用：Person.setName(“xxx”).setAge(“19”)；

8、@Slf4j

自动生成该类的 log 静态常量，要打日志就可以直接打，不用再手动 new log 静态常量了

这里有一个例子，一个鉴权注解的定义。在这里，用到了@Target、@Retention和@Doucumented

@Target

@Target用来表示注解作用范围，超过这个作用范围，编译的时候就会报错。

枚举类如上所示，其中包括

枚举变量 作用域
Type 类或接口
FIELD 域
METHOD 方法
PARAMETER 参数
CONSTRUCTOR 构造方法
LOCAL_VARIABLE 局部变量
ANNOTATION_TYPE 注解类型（被interface修饰的类型）
PACKAGE 用于记录java文件的package信息
这里这个@Inner注解可以使用在方法和类或接口上。

@Retention

@Rentention定义了该Annotation被保留的时间长短

@Rentention由RetentionPolicy表示需要在什么级别保存该注解信息，用于描述注解的生命周期。

枚举变量 有效范围
SOURCE 在源文件中有效
CLASS 在class文件中有效
RUNTIME 在运行时有效

这边的@Inner注解表示在运行时有效。

@Documented

Documented注解表明这个注释是由 javadoc记录的，在默认情况下也有类似的记录工具。 如果一个类型声明被注释了文档化，它的注释成为公共API的一部分。

@Component

Component泛指组件，当组件不好归类的时候，我们可以使用这个注解进行标注

二分法

今日做LeetCode剑指Offer，做到了二分法的题目，写起来老是自己绕进去，看一看整理一下基本模板。

例题

剑指 Offer 53 - II. 0～n-1中缺失的数字

正所谓，看到递增数组，就要用二分法，用遍历方法等于白做。

我的第一思路是，取low=0,high=n-1，然后开始二分mid = low + (high - low) / 2，验证如果nums[mid] != mid，则证明这个数字在low到mid范围内，否则是在mid到high范围内。

二分法最蛋疼的一点就是这个边界，我这个解法在每次都是令low或high=mid，然后结束循环的条件是low < high -1，避免low = high -1导致mid无限等于low。最终就会出现low与high差2，比如对一个递增数字找到某个数字时，low与high中间的值就是需要的结果。这个题目实际上应该等于nums[mid]-1，但是我这个mid放循环里了，所以就用的nums[low]+1。

LeetCode题解区我看到了一个非常简洁的二分

属实牛批。

模板

模板1

二分将范围分成(low,mid)和(mid+1,high)，更新操作时令low = mid + 1，high = mid，判断条件使用low < high即可，这个是用得最广泛的，最终low与high塌成一个点。

模板2

二分将范围分成(low,mid-1)和(mid,high)时，更新操作时令low = mid， high = mid - 1，mid应该设为(low + high + 1) / 2

模板3

也就是我用的那个，将范围分成(low,mid)和(mid,high)，边界已经排除，这方法就是得使劲排除边界条件。

这里有一个蓄水池算法，是我抄的，学到了学到了，用在不知道总数是多少的情况。**抽奖算法**

所谓蓄水池算法，从头来说可以这样形容：

（1）来到第一个节点，池子里只有一个数字，这时候随机到这个数字的概率必为100%

（2）来到第二个节点，这时候池子里有两个数字，这时候随机到这两个数字中其中一个的概率为50%

（3）来到第三个节点，这时候池子里有三个数字，这时候随机到第三个数字的概率为33%，随机到前两个数字的概率均为66%*50%=33%，三个数字概率相同。

以此类推，链表遍历完以后，能够保证随机到任意一个数字的概率均相等。
